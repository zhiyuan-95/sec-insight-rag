## Question 1 - Directory Structure

Question

    Should we change the overall file and directory structure based on the fact that we have ingestion, derivation, analysis, and LLM reasoning?

### Core Issue

    The issue is how to split the system so each part has a clear job.

    This project is not just one analysis script. It has several different kinds of work:

    - SEC/XBRL ingestion
    - metrics normalization
    - indicator calculation
    - financial data analysis
    - semantic filing analysis
    - LLM reasoning

    If these parts are mixed together, the system will be harder to test, debug, and trust.

### Recommendation

    Yes. Keep separate folders for ingestion, processing, indicators, financial data analysis, retrieval, and LLM reasoning.

    The structure in `proposal.md` and `agents.md` is a good direction:

    ```text
    src/
      config/
      api/
      ingestion/
      processing/
      indicators/
      analytics/
      retrieval/
      analyze/
      storage/
      evaluation/
    ```

    I would keep this structure.

### Why

    Each part does different work:

    - `ingestion/` gets SEC data and filings.
    - `processing/` cleans and normalizes XBRL facts.
    - `indicators/` calculates derived financial metrics.
    - `analytics/` performs financial data analysis.
    - `retrieval/` finds relevant filing text.
    - `analyze/` uses Gemini to explain the evidence.

    This makes the source of each result easier to trace.

### Tradeoff

    Better:

    - Easier to test each part.
    - Easier to find bugs.
    - Easier to trace final answers back to source data.
    - Easier to add RAG and LLM reasoning later.

    Worse:

    - More files.
    - More module boundaries.
    - You need clear data contracts between steps.

### Concrete Project Example

    For the question "Why did revenue change?":

    - `ingestion/` gets SEC company facts and filings.
    - `processing/` normalizes revenue facts by period and unit.
    - `indicators/` calculates revenue growth.
    - `analytics/` checks trends or unusual changes.
    - `retrieval/` finds MD&A text about revenue.
    - `analyze/` uses Gemini to explain the answer with evidence.

    This is cleaner than one large file doing everything.

### Suggested Next Step

    Keep the proposed folder structure.

    Before coding, write a short responsibility note for each folder so the boundaries stay clear.

### Comments And Further Discussion

### Decision And Implementation

    Decision: Keep the layered backend structure already described in `proposal.md`.

    Implementation: Maintain separate modules for ingestion, processing, indicators, analytics, retrieval, LLM analysis, storage, workflows, and API routes. Do not collapse these responsibilities into one large module. If the structure changes later, update the current architecture documentation instead of rewriting old milestone notes.

## Question 2 - Main Goal And Evidence Separation

Question

    Why is the goal written as "separate reported facts, calculated indicators, financial data analysis results, semantic filing analysis, and model-generated interpretations so users can trust where each conclusion comes from"?

    Shouldn't the main goal just be evidence-based financial analysis?

### Core Issue

    The issue is the difference between the main product goal and the internal design rule.

    The main goal can be simple:

    ```text
    Build an evidence-based financial analysis system.
    ```

    But to make that goal real, the system must separate different evidence types.

### Recommendation

    Use "evidence-based financial analysis" as the main goal.

    Use separation of evidence types as the architecture rule.

    That means the system should keep these things separate and traceable:

    - reported facts
    - calculated indicators
    - financial data analysis results
    - semantic filing evidence
    - LLM interpretations

### Why

    These are not the same kind of information.

    A reported fact comes from SEC/XBRL data.

    A calculated indicator comes from a formula.

    A financial data analysis result comes from deterministic analysis code.

    Semantic filing evidence comes from filing text.

    An LLM interpretation is the model's explanation based on the evidence.

    If they are mixed together, the answer may sound confident even when the evidence is weak.

### Tradeoff

    Better:

    - More trustworthy answers.
    - Easier to audit.
    - Easier to cite evidence.
    - Easier to prevent unsupported causal claims.

    Worse:

    - Responses may be more structured.
    - The backend needs more metadata.
    - Prompts need to be stricter.

### Concrete Project Example

    Bad answer:

    ```text
    Revenue declined because demand was weak.
    ```

    Better answer:

    ```text
    Revenue declined based on SEC XBRL facts.
    Revenue growth was negative based on the calculated formula.
    Financial data analysis shows this decline continued across multiple periods.
    The filing mentions lower sales volume.
    So weaker demand may be a reason, but only if the filing supports that claim.
    ```

    The second answer is better because it separates fact, calculation, analysis, filing evidence, and interpretation.

### Suggested Next Step

    Use this wording:

    ```text
    Main goal:
    Build an evidence-based financial analysis system.

    Design rule:
    Keep reported facts, calculated indicators, financial data analysis results,
    semantic filing evidence, and LLM interpretations separate and traceable.
    ```

### Comments And Further Discussion

### Decision And Implementation

    Decision: Use "evidence-based financial analysis" as the simple product goal, and use evidence separation as the internal architecture rule.

    Implementation: Keep reported facts, derived indicators, deterministic analysis results, semantic filing evidence, and LLM interpretations as separate inputs and outputs. Prompt templates, response schemas, and stored analysis outputs should preserve those labels instead of blending everything into one explanation.

## Question 3 - Pipeline Flow Versus One Big File

Question

    Is it better to combine data ingestion, metrics normalization, and indicator derivation into one file?

    The reason I ask is that ingestion, normalization, and processing seem like a data processing pipeline. Each step contains or depends on the result of the previous step, so I do not think a separate pipeline file is needed.

### Core Issue

    The issue is the difference between data flow and code ownership.

    You are right that ingestion, normalization, and indicator derivation happen in order. Each step uses the result from the previous step.

    That means the system has a pipeline flow. But it does not automatically mean the project needs a separate `pipelines/` folder or a separate pipeline file at the beginning.

### Recommendation

    Do not combine ingestion, normalization, and indicator derivation into one large file.

    Also, do not add a separate pipeline file unless the orchestration logic becomes complex enough to deserve one.

    A simple v1 structure can be:

    ```text
    src/
      ingestion/
      processing/
      indicators/
      analytics/
      retrieval/
      analyze/
    ```

    The important rule is:

    ```text
    Keep the logic for each stage separate.
    Let one caller run the stages in order.
    ```

    That caller could be an API handler, an application service, or later a dedicated pipeline file if the flow grows.

### Why

    Each step has a different responsibility:

    - Ingestion handles SEC requests, downloads, retries, throttling, and raw responses.
    - Normalization handles XBRL concepts, units, periods, forms, and missing data.
    - Indicator derivation handles formulas like revenue growth, margin, debt ratio, and free cash flow.

    These should be tested separately.

    The fact that one step depends on the previous step is normal. For example:

    ```text
    ingestion result -> processing result -> indicator result -> analytics result
    ```

    But dependency in the data flow should not mean all code belongs in one file.

### Tradeoff

    Better:

    - Cleaner files.
    - Easier unit tests.
    - Easier debugging.
    - Easier data lineage.
    - Fewer folders than adding a `pipelines/` layer too early.

    Worse:

    - The API handler or service may start to contain orchestration code.
    - If the workflow grows, you may later need to extract a dedicated pipeline module.
    - You still need clear data objects passed between steps.

### Concrete Project Example

    If the SEC company facts response has duplicate revenue values, that is a `processing/` concern.

    If free cash flow is calculated as operating cash flow minus capital expenditures, that is an `indicators/` concern.

    If the system detects that revenue declined for three periods, that is an `analytics/` concern.

    If MD&A explains that revenue changed because of sales volume, that is a `retrieval/` and semantic filing analysis concern.

    If Gemini explains all of this in plain English, that belongs in `analyze/`.

    Putting all of this into one file would make the system harder to audit.

### Suggested Next Step

    Do not create a separate `pipelines/` folder yet unless the workflow becomes hard to read.

    For now, keep the stage logic separate and let one higher-level function call the steps in order.

    If that higher-level function becomes too large, then extract it into a dedicated pipeline module later.

### Comments And Further Discussion

### Decision And Implementation

    Decision: Do not combine ingestion, normalization, and indicator derivation into one file. Also do not add a dedicated pipeline folder until orchestration becomes large enough to justify it.

    Implementation: Keep stage logic in `src/ingestion/`, `src/processing/`, `src/indicators/`, and `src/analytics/`. Use a thin caller or workflow function to run the stages in order. Extract a separate pipeline/workflow module only when the orchestration code becomes hard to read or test.

Question

	we are going to keep latest 5-year of 10k and 12 quarters of 10q in the systems. which means our systems is going to do some deque kind of operation first in first out. and I want to ingest xbrl facts into my rag system I want to add all the xbrl fact available of the company in the system. but then length of the historical data of the company in the system would depend on the date of the ingestion of the company since the eariliest traceable date will roll forward. how should I handle it?

### Core Issue

    The real issue is the difference between retention, retrieval scope, and source-of-truth storage.

    You want the system to feel current and manageable by keeping the latest 5 years of 10-K data and 12 quarters of 10-Q data available for normal use.

    But you also want to ingest all available XBRL facts for a company so the RAG and analysis system has a broad historical record.

    Those two goals conflict only if "keep latest 5 years / 12 quarters" means physically deleting older raw facts.

    If older raw XBRL facts are deleted, then the earliest traceable date depends on when the company was first ingested. That makes analysis inconsistent across companies and weakens auditability.

### Recommendation

    Do not implement this as a physical FIFO deletion of raw XBRL facts.

    Use a two-layer policy instead:

    ```text
    Raw XBRL fact archive:
      Keep all normalized XBRL facts available from SEC companyfacts.
      This is the source-of-truth history.

    Active analysis window:
      Default to latest 5 fiscal years of 10-K data and latest 12 quarters of 10-Q data.
      This is the normal inspection, metrics, analytics, and RAG scope.
    ```

    In other words:

    ```text
    Store broadly.
    Retrieve narrowly by default.
    Preserve traceability.
    ```

    The "deque" behavior should be a query/filtering policy, not a destructive storage policy.

### Why

    I inspected the current and planned storage direction:

    - `src/storage/database.py` currently creates `raw_xbrl_facts`.
    - `src/storage/facts_repository.py` upserts raw facts and retrieves them by CIK/concept.
    - `plan2_5.txt` adds planned `companies`, `filings`, and `financial_metrics` tables.

    That structure is already pointing in the right direction.

    `raw_xbrl_facts` should remain the complete auditable layer. It is compact compared with filing HTML and is useful for long-term trend checks, restatement review, and future reprocessing.

    The latest 5 years / 12 quarters rule should apply to the default business-facing layer:

    - `financial_metrics`
    - indicator calculation
    - deterministic analytics
    - RAG evidence selection
    - API responses unless the user asks for older history

    This avoids making the system's historical range depend on the original ingestion date.

    For example, if AAPL is first ingested in 2026, SEC companyfacts may still include many older years. The system should keep those raw facts. But the default "company snapshot" can still focus on FY2021-FY2025 and the latest 12 quarters.

    This gives you both:

    - current, manageable answers for users
    - deeper history for audit, rebuilds, and future analysis

### Tradeoff

    Better:

    - Default answers stay focused on recent financial performance.
    - Raw XBRL evidence remains auditable.
    - Different companies have more consistent historical coverage.
    - You can rebuild base metrics or indicators later without re-fetching everything.
    - You avoid losing data because of a rolling-window mistake.

    Worse:

    - The local SQLite database stores more raw facts.
    - Retrieval code needs an explicit active-window filter.
    - You need to distinguish "stored history" from "default analysis window."
    - If filing HTML storage becomes large, you may need a separate file-retention policy.

### Concrete Project Example

    Use this model:

    ```text
    raw_xbrl_facts
      all normalized XBRL facts available from SEC companyfacts
      not physically FIFO-deleted by the 5-year / 12-quarter rule

    filings
      filing metadata for ingested filings
      can mark whether a filing is inside the active analysis window

    financial_metrics
      mapped base metrics from raw_xbrl_facts
      default queries return latest 5 annual periods and latest 12 quarterly periods

    retrieval / RAG
      does not embed every XBRL fact as text by default
      retrieves structured facts from SQLite and converts only relevant facts into answer evidence
    ```

    For a revenue question:

    ```text
    User asks: Why did revenue change?

    System default:
      retrieve recent revenue metrics from financial_metrics
      use latest 5 annual periods or latest 12 quarters
      trace each metric to raw_xbrl_facts and filings
      retrieve filing text only for relevant filings
      let Gemini explain using labeled evidence

    System audit mode:
      query older raw_xbrl_facts if needed
      show original XBRL concept, unit, period, accession number, and quality flags
    ```

    This is better than putting all XBRL facts into a vector index. XBRL facts are structured data, so SQLite is the better retrieval system for them. Vector retrieval is better for filing text such as MD&A, Risk Factors, Business, and notes.

### Core Issue

    The real issue is that "storage problem" covers more than downloaded filing HTML.

    The initial Question 6 answer was directionally correct but incomplete. It separated durable data from active-window data, but it did not explicitly say that retrieval artifacts created from old HTML also need to follow the active-window rule.

    If the system deletes old filing HTML but keeps old parsed chunks, embeddings, and vector index entries, then storage still grows and retrieval may return evidence from filings that are no longer supposed to be active.

    So the storage policy must cover all filing-derived artifacts, not only the original HTML files.

### Recommendation

    Use a three-layer storage policy:

    ```text
    Durable structured layer:
      Keep long term.
      Includes company metadata, filing metadata, raw XBRL facts, and source identifiers.

    Active filing evidence layer:
      Keep only for the active window by default.
      Includes downloaded filing HTML and extracted filing text.

    Active retrieval index layer:
      Keep only for the active window by default.
      Includes parsed chunks, embeddings, vector index entries, and chunk metadata used for semantic retrieval.
    ```

    Amendment to the initial Question 6 recommendation:

    ```text
    Keep filing metadata and raw XBRL facts long term.
    Keep downloaded filing HTML, parsed chunks, embeddings, and vector index entries only for active-window filings by default.
    Delete or deactivate all retrieval artifacts tied to filings outside the active window.
    Rebuild older filing evidence only when the user explicitly asks for older history.
    ```

    So the rule is not just:

    ```text
    delete old HTML
    ```

    The rule is:

    ```text
    delete or deactivate the whole filing-evidence bundle for filings outside the active window
    ```

### Why

    Filing HTML is only the first storage object.

    A retrieval pipeline usually creates several derived artifacts:

    - downloaded HTML
    - cleaned filing text
    - section metadata
    - parsed chunks
    - embeddings
    - vector index records
    - chunk-to-filing references

    If older HTML is deleted but old embeddings remain, the system has two problems:

    - storage is still consumed by inactive filings
    - retrieval may surface stale evidence from outside the intended active window

    That creates a mismatch between what the product says it keeps and what the RAG system can actually retrieve.

    The better design is to keep compact, auditable data permanently and keep bulky retrieval artifacts only for the active evidence window.

    `raw_xbrl_facts` should stay because XBRL facts are structured, compact, and useful for rebuilding metrics. Filing HTML, chunks, embeddings, and indexes are larger and easier to rebuild from SEC accession metadata.

### Tradeoff

    Better:

    - Storage stays bounded as more companies are added.
    - RAG retrieval stays aligned with the active 5-year / 12-quarter policy.
    - Raw financial facts remain auditable.
    - Older filing evidence can still be rebuilt when needed.
    - The system avoids keeping orphan embeddings for deleted filing files.

    Worse:

    - Retrieval cleanup becomes more complex than deleting files.
    - The system needs metadata linking chunks and embeddings back to filing accession numbers.
    - Older filing Q&A may require a rebuild step before answering.
    - Deleting vector records can be harder depending on the vector store.

### Concrete Project Example

    Use this lifecycle:

    ```text
    Filing inside active window:
      keep filing metadata
      keep raw XBRL facts
      keep downloaded HTML
      keep parsed chunks
      keep embeddings
      keep vector index entries

    Filing outside active window:
      keep filing metadata
      keep raw XBRL facts
      delete downloaded HTML by default
      delete parsed chunks by default
      delete embeddings by default
      delete or deactivate vector index entries by default
    ```

    The important foreign key is the filing accession number:

    ```text
    filings.accession_number
      -> downloaded file path
      -> filing chunks
      -> embeddings
      -> vector index entries
    ```

    When a filing leaves the active window, cleanup should be driven by accession number:

    ```text
    cleanup_filing_evidence(accession_number)
      -> delete HTML file
      -> delete chunk rows
      -> delete embedding rows
      -> delete/deactivate vector index records
      -> keep filing metadata
      -> keep raw_xbrl_facts
    ```

    If the user later asks for older history:

    ```text
    older filing requested
      -> use filing metadata/accession number
      -> re-download filing HTML if needed
      -> re-parse chunks
      -> rebuild embeddings/index entries
      -> answer with explicit note that older evidence was rebuilt on demand
    ```

### Suggested Next Step

    Update Plan 2.5 or the future retrieval milestone with this policy:

    ```text
    Active-window retention applies to filing evidence artifacts:
    downloaded HTML, parsed text, chunks, embeddings, and vector index entries.

    Long-term retention applies to durable structured artifacts:
    company metadata, filing metadata, raw XBRL facts, and source identifiers.
    ```

    When the retrieval layer is implemented, every chunk and embedding should store:

    ```text
    cik
    accession_number
    form_type
    filing_date
    fiscal_year
    fiscal_period
    active_window flag
    source file path or source URL
    ```

    This makes cleanup deterministic and prevents stale retrieval results.
