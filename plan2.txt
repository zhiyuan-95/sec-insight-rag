Milestone 2 Design: SEC/XBRL Ingestion and Normalization

Task Restatement
Build the single-company SEC/XBRL ingestion and normalization capability. This milestone should resolve a ticker to a CIK, retrieve SEC submissions and companyfacts JSON, download latest 10-K and 10-Q filing documents, normalize XBRL companyfacts into auditable raw financial fact records, store normalized facts in SQLite, and add a thin company-level orchestration function exported from `src.ingestion`. It should not calculate derived indicators, run financial analytics, build retrieval indexes, call Gemini, generate RAG answers, or add frontend code.

Success Criteria
- A ticker can be resolved to a zero-padded 10-digit CIK.
- SEC HTTP requests include the configured SEC_USER_AGENT and use bounded retry/throttling behavior.
- Company submissions JSON and companyfacts JSON can be retrieved for a CIK.
- The latest available 10-K and 10-Q primary filing documents can be selected and downloaded.
- Common GAAP financial concepts can be extracted from companyfacts JSON.
- XBRL facts are normalized by concept, taxonomy, unit, period, fiscal year, fiscal period, form type, filing date, accession number, frame, and source metadata.
- Raw SEC values are preserved, with parsed numeric values stored separately when possible.
- Duplicate, missing, malformed, unsupported, or ambiguous facts are flagged instead of silently guessed.
- Normalized raw facts can be stored in SQLite using STOCK_SQL_DB_PATH.
- `ingest_company` can be imported from `src.ingestion` and can orchestrate the full company ingestion flow without duplicating lower-level logic.
- Tests use mocked SEC responses, local fixtures, and temporary SQLite databases; they do not require live network access.

Important Edge Cases
- SEC_USER_AGENT is missing or blank.
- Ticker input has lowercase letters, whitespace, punctuation, or an unknown symbol.
- SEC ticker mappings contain numeric CIK values that need zero padding.
- SEC returns 403, 404, 429, or 5xx responses.
- SEC returns invalid JSON or payloads missing expected fields.
- Submissions data has no recent 10-K or 10-Q filing.
- Filing metadata is missing accession number, primary document, filing date, or form type.
- Filing accession numbers include dashes, while SEC archive URLs require dashless accession directories.
- The target filing file already exists locally.
- Companyfacts payloads are missing cik, entityName, facts, taxonomy, concept, units, or fact arrays.
- Concepts appear under different taxonomies, especially us-gaap and dei.
- A concept has multiple units, such as USD, shares, USD/shares, or pure.
- Duration facts have start and end dates; instant facts only have end dates.
- Annual and quarterly facts for the same concept may overlap.
- The same concept, unit, period, form, accession, fiscal year, and fiscal period may appear more than once.
- Numeric values may be integers, floats, negative values, strings, or null.
- Companyfacts may include old periods, restatements, amended filings, 8-K facts, or other unsupported forms.
- SQLite database directories may not exist yet.
- Re-running ingestion for the same company should not create uncontrolled duplicate fact rows.

Dependency Design
Avoid adding new runtime dependencies unless implementation proves the standard library and existing dependencies are insufficient.

Use the standard library for:
- urllib.request for SEC HTTP requests.
- json for response parsing and quality flag storage.
- pathlib for path handling.
- time for request throttling.
- dataclasses for normalized records and workflow result objects.
- datetime for date validation.
- decimal.Decimal for parsed numeric values.
- sqlite3 for local persistence.

Do not use Gemini, LlamaIndex, pandas, or an ORM in this milestone.

Configuration Design
Use the existing Settings object from src/config/settings.py.

Required for live SEC ingestion:
- sec_user_agent
- stock_filings_base_dir
- stock_sql_db_path

Behavior:
- Milestone 1 may allow missing SEC_USER_AGENT so the app can start.
- Live SEC ingestion should reject missing or blank SEC_USER_AGENT before making requests.
- Create parent directories for downloaded filings and SQLite databases when needed.
- Do not hardcode user-specific paths.
- Do not print or log secrets, API keys, or full raw SEC payloads.

Module Responsibility Design
Keep implementation layers separate even though the product operation is one combined ingestion workflow.

src/ingestion:
- Own SEC-facing concerns only.
- Resolve ticker to CIK.
- Fetch submissions and companyfacts JSON.
- Select latest 10-K and 10-Q filings.
- Download primary filing documents.
- Handle SEC request headers, retries, throttling, and SEC-specific errors.

src/processing:
- Own XBRL/companyfacts normalization only.
- Extract selected concepts.
- Normalize units, periods, fiscal years, fiscal periods, forms, filing dates, accession numbers, frames, and quality flags.
- Preserve raw SEC values and parsed Decimal values separately.
- Do not fetch SEC data and do not write SQLite rows directly.

src/storage:
- Own SQLite persistence only.
- Initialize local tables.
- Upsert normalized raw facts without uncontrolled duplicate rows.
- Retrieve normalized facts by CIK and optional concept filters.

src/ingestion/company.py:
- Own the thin company-level ingestion orchestration function.
- Call existing ingestion, processing, and storage functions in order.
- Return a compact result object for API routes and scripts.
- Do not duplicate SEC HTTP, normalization, or SQL logic.

- parse_sec_date(value) -> date | None
- classify_period(start_date, end_date) -> str

Storage APIs already planned or implemented:

- RawFactRepository.initialize() -> None
- RawFactRepository.upsert_facts(facts) -> int

Company ingestion API:
- CompanyIngestionResult
  - ticker: str
  - cik: str
  - filings: tuple[FilingMetadata, ...]
  - downloaded_filings: tuple[Path, ...]
  - normalized_fact_count: int
  - stored_fact_count: int
  - warnings: tuple[str, ...]

- ingest_company(ticker: str, settings: Settings) -> CompanyIngestionResult
  - Load SEC ticker mapping.
  - Resolve ticker to CIK.
  - Retrieve company submissions.
  - Retrieve companyfacts.
  - Select and download latest 10-K and 10-Q primary documents.
  - Normalize companyfacts.
  - Initialize SQLite storage.
  - Upsert normalized facts.
  - Return a structured result.

Data Integrity Rules
- Treat raw SEC fixtures as immutable test inputs.
- Preserve raw SEC values in normalized facts.
- Store parsed numeric values as Decimal text to avoid floating-point drift.
- Store quality flags as JSON text in SQLite.
- Do not delete existing facts unless a future refresh policy explicitly adds that behavior.
- Do not calculate derived indicators in this milestone.
- Do not infer values when facts are ambiguous or incomplete.

Testing Design
Use local fixtures, mocked SEC responses, and temporary SQLite databases.

SEC ingestion tests:
- Reject blank SEC_USER_AGENT.
- Add User-Agent, Accept, and Accept-Encoding headers.
- Parse JSON responses.
- Raise clear errors for HTTP failures and invalid JSON.
- Retry only retryable SEC status codes.
- Parse SEC company_tickers-style JSON.
- Normalize ticker symbols and zero-pad CIKs.
- Resolve lowercase and whitespace-padded ticker input.
- Build submissions and companyfacts URLs.
- Validate submissions and companyfacts payload shapes.
- Extract latest 10-K and 10-Q metadata.
- Build SEC archive document URLs.
- Download filing bytes below the configured filings directory.
- Reuse existing downloaded files.

XBRL normalization tests:
- Extract common us-gaap concepts from companyfacts fixtures.
- Preserve CIK, entity name, taxonomy, concept, unit, form, fiscal year, fiscal period, accession number, and dates.
- Parse numeric values into Decimal while preserving raw values.
- Handle instant facts and duration facts.
- Filter by requested concepts.
- Filter to 10-K and 10-Q by default.
- Flag unsupported forms, duplicate facts, ambiguous units, missing values, non-numeric values, missing end dates, and inconsistent periods.

SQLite repository tests:
- Create the raw_xbrl_facts table.
- Insert normalized facts.
- Upsert duplicate facts without multiplying rows.
- List facts by CIK.
- Filter facts by concept.
- Round-trip quality flags and Decimal values.

Company ingestion tests:
- Mock lower-level SEC calls and confirm `ingest_company` calls them in order.
- Use fixture companyfacts and a temporary SQLite database to confirm the function stores normalized facts.
- Confirm the function returns ticker, CIK, filing metadata, downloaded filing paths, normalized count, stored count, and warnings.
- Confirm the function rejects missing SEC_USER_AGENT before making SEC calls.

Commands to verify:
- uv run pytest

Manual Acceptance Checks
With a valid SEC_USER_AGENT in config.env:
- Call `ingest_company("AAPL", settings)`.
- Confirm it resolves AAPL to a CIK.
- Confirm it retrieves submissions and companyfacts.
- Confirm it downloads the latest 10-K and 10-Q primary documents.
- Confirm it normalizes companyfacts into raw financial fact records.
- Confirm it stores normalized facts in SQLite.
- Query stored facts for one CIK and a concept such as Revenues.
- Confirm no indicators, analytics, retrieval indexes, Gemini calls, RAG answers, CSV exports, or frontend code are involved.

Out of Scope
- Derived indicators such as margins, ratios, growth, free cash flow, or cash conversion.
- Period-over-period analysis.
- Benchmark or peer comparison.
- CSV export routes.
- FastAPI ingestion or metrics routes.
- Filing text parsing or vector indexing.
- LlamaIndex integration.
- Gemini model integration.
- RAG analysis.
- Frontend code.

Implementation Order
1. Keep the existing SEC ingestion modules as the SEC-facing layer.
2. Keep the existing XBRL normalization modules as the processing layer.
3. Keep the existing SQLite repository modules as the storage layer.
4. Add `src/ingestion/company.py` with `CompanyIngestionResult` and `ingest_company`.
5. Export `CompanyIngestionResult` and `ingest_company` from `src/ingestion/__init__.py`.
6. Add company ingestion tests using mocked SEC calls, local companyfacts fixtures, and temporary SQLite storage.
7. Run `uv run pytest`.

Assumptions
- The old Plan 2 and Plan 3 are now one combined milestone because users expect "ingest a company" to make normalized facts available.
- The code should keep ingestion, processing, storage, and workflow responsibilities separate even though the milestone wording is combined.
- The company-level ingestion function is an orchestration layer, not a place to duplicate SEC HTTP, XBRL normalization, or SQL persistence logic.
- Indicator calculation starts in the next milestone.
- Retrieval and RAG remain later milestones.
